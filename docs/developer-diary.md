# 开发者日记

## 项目名称

学院学生综合服务与党团管理平台

## 记录目的

本文档用于记录当前仓库的开发推进过程、阶段性成果、关键设计决策和后续待办，便于项目成员交接、课程答辩说明和后续继续开发。

## 2026-04-19  CRUEL

### 1. 需求整理与文档转换

本阶段首先处理了原始需求文档，将 `学院学生综合服务与党团管理平台_软件需求规格说明书_已填模板(1).docx` 中的内容提取并整理为 Markdown，写入项目 `README.md`。

本次处理包含：

- 提取文档正文、表格和章节结构
- 将原有软件需求规格说明书改写为仓库内可直接阅读的 Markdown 文档
- 保留需求中的目标、角色、功能模块、性能要求、交付要求与原型范围

产出文件：

- `README.md`

### 2. 初步原型开发

在仓库尚无前后端工程骨架的情况下，为了尽快把需求落成可演示成果，先完成了一版纯静态页面原型。

本阶段实现内容：

- 创建平台入口页
- 创建学生端页面
- 创建管理端页面
- 抽取共享样式文件
- 抽取共享脚本与模拟数据
- 实现基础导航切换、知识库筛选、通知已读、证明申请等前端交互

产出文件：

- `index.html`
- `student.html`
- `admin.html`
- `styles.css`
- `scripts/app.js`

### 3. 首版实现范围

学生端已覆盖：

- 首页聚合视图
- 知识库查询
- 党团进度展示
- 通知中心
- 证明申请记录

管理端已覆盖：

- 业务概览
- 审批处理列表
- 知识库管理视图
- 学生画像视图
- 审计日志视图

### 4. 本阶段决策

- 由于当前仓库没有现成技术栈和依赖安装流程，优先使用原生 HTML、CSS、JavaScript 实现可直接打开运行的静态原型
- 原型重点放在信息架构、业务边界和交互演示，不提前绑定具体后端框架
- 将学生端和管理端拆分为独立页面，便于后续迁移为前后端分层架构

建议后续按以下顺序推进：

4. 将当前静态原型改造为真正消费接口的数据页面
